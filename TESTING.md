# Testing

## Prerequisites

All commands require [`just`](https://github.com/casey/just), [`shellcheck`](https://www.shellcheck.net/), the [`devcontainer` CLI](https://github.com/devcontainers/cli), and Docker. These are pre-installed in the devcontainer.

---

## Running tests locally (x86)

| Command               | What it does                                                   |
| --------------------- | -------------------------------------------------------------- |
| `just`                | Lint and run all tests (default)                               |
| `just lint`           | Shellcheck all `.sh` files + Prettier format check             |
| `just test`           | Run all feature tests on x86 (Debian + Alpine where supported) |
| `just test-<feature>` | Test a single feature, e.g. `just test-act`                    |

`just test` runs autogenerated (`test.sh`) and scenario (`scenarios.json`) tests for every feature via the `devcontainer` CLI. Each feature is tested against Debian Bookworm; most are also tested against Alpine 3.20 (see the feature table in README.md for coverage).

---

## Running tests locally (arm64)

arm64 tests use Docker BuildKit to build and run feature images under QEMU emulation. Only autogenerated (`test.sh`) tests are run locally — scenario tests for arm64 are covered by CI.

**One-time setup per container restart:**

```bash
just setup-qemu
```

This registers QEMU binfmt handlers in the kernel so arm64 containers can run on x86. The registration persists until the devcontainer is restarted.

| Command                     | What it does                                               |
| --------------------------- | ---------------------------------------------------------- |
| `just test-arm64`           | Run all feature autogenerated tests on arm64               |
| `just test-<feature>-arm64` | Test a single feature on arm64, e.g. `just test-act-arm64` |

arm64 test images are cached by Docker after the first build. Subsequent runs are fast unless the feature source changes.

**Alpine exclusions on arm64:** `kiro-cli` and `nvidia-container-toolkit` are Debian-only on arm64, matching their x86 alpine exclusions.

## CI testing

CI runs on every push and pull request via [`.github/workflows/ci.yml`](.github/workflows/ci.yml). All test jobs run on both `ubuntu-latest` (x86) and `ubuntu-24.04-arm` (native arm64 — no emulation).

| CI job               | What it runs                            | Runners                     |
| -------------------- | --------------------------------------- | --------------------------- |
| `lint`               | shellcheck + prettier                   | x86 only (arch-independent) |
| `test-autogenerated` | `test.sh` for each feature × base image | x86 + arm64                 |
| `test-scenarios`     | `scenarios.json` tests for each feature | x86 + arm64                 |
| `test-global`        | `test/_global/` scenarios               | x86 + arm64                 |

### Test types

**Autogenerated tests** (`test.sh`) — verify the feature installs correctly and the main binary is functional. Run via `devcontainer features test --skip-scenarios`.

**Scenario tests** (`scenarios.json` + `<scenario>.sh`) — test specific feature options (e.g. a pinned version, a non-root user, Alpine musl). Each scenario entry specifies a base image and feature options; the matching `.sh` script runs inside the built container. Run via `devcontainer features test --skip-autogenerated`.

**Global scenarios** (`test/_global/`) — tests that span multiple features or test cross-feature interactions.

## Simulating CI locally with `act`

| Command             | What it does                                    |
| ------------------- | ----------------------------------------------- |
| `just act-ci`       | Run the full CI workflow locally                |
| `just act-validate` | Run the validate workflow locally               |
| `just act-dry`      | Dry-run both workflows (no containers launched) |
| `just act-all`      | Run both workflows locally                      |

`just act-dry` is useful for checking that a workflow YAML change is structurally valid before pushing.
